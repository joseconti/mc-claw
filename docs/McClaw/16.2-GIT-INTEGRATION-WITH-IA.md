# 16.2 - Git Integration with AI (Deep Interaction Layer)

## Status: Planned

## Goal

Transform the Git section from a **passive repository viewer** into a **fully interactive AI-powered Git workstation** where the user can do virtually anything with their repositories through the AI chat and through direct UI actions. The AI must be deeply integrated at every level: every element the user sees should be actionable, and the AI should be capable of performing complex multi-step Git workflows autonomously.

This document extends `16-GIT-INTEGRATION.md` and focuses exclusively on the AI interaction layer — what the AI can do, how it does it, and how the UI connects to the AI at every point.

---

## Current State (What Exists)

The Git section currently has:
- Repository list from GitHub/GitLab connectors (read-only)
- Repository detail view with tabs: Code, Issues, Pull Requests, Commits
- File tree viewer with syntax highlighting (15+ languages)
- Branch selector
- Git context chip in the chat input bar
- `GitService` actor with read AND write operations defined (but write operations are never called)
- `GitActionConfirmationCard` component (exists but is never used — it is an orphan component)
- `@git()` intercept defined in PromptEnrichmentService (but not fully wired to execution)

### Critical Gaps

1. **No write operations work.** GitService has `commit()`, `push()`, `createBranch()`, etc., but nothing in the UI or chat triggers them.
2. **The confirmation card is orphaned.** `GitActionConfirmationCard` exists as a SwiftUI view but is never instantiated anywhere.
3. **The chat does not intercept AI responses.** When the AI responds with `@git(...)`, there is no code that detects this, executes the command, and feeds the result back.
4. **No contextual actions on UI elements.** PRs, issues, commits, and files are display-only. No "Ask AI about this" or "Review this PR" buttons anywhere.
5. **No mechanism for the AI to propose write actions.** The AI cannot say "I want to create a branch" and have McClaw show a confirmation dialog.

---

## Design Principle: Every Element Is Actionable

The core principle of this integration is: **anything the user can see, the user can act on through the AI, and the AI can act on autonomously.**

This means:
- Every PR row has an "Ask AI" button
- Every issue row has an "Ask AI" button
- Every commit row has an "Explain" action
- Every file has "Ask about this file" and "Ask about selected lines"
- The AI can propose write actions, and McClaw shows confirmation cards
- The AI can chain multiple operations in sequence (clone → branch → commit → push → create PR)

## 1. Chat ↔ Git Execution Pipeline

This is the most critical missing piece. The full pipeline must work as follows:

### 1.1 User Sends Message with Git Context

```
User types: "What changed this week in this repo?"

ChatViewModel:
  1. Reads gitContext (repo: acme-api, branch: main, platform: github)
  2. Calls PromptEnrichmentService.buildGitContextHeader(gitContext)
  3. Prepends header to user message
  4. Sends enriched message to CLI via CLIBridge
```

The context header that PromptEnrichmentService must build:

```
[McClaw Git Context]
Platform: GitHub
Repository: joseconti/acme-api (https://github.com/joseconti/acme-api)
Branch: main
Local path: /Users/joseconti/Developer/acme-api
Cloned locally: yes

Available platform actions (via @fetch):
  READ: list_repos, list_branches, list_issues, list_prs, get_pr_diff, list_commits, list_releases, search_code, get_contents
  WRITE: create_issue, close_issue, create_comment, create_pr, merge_pr, create_release

Available local git actions (via @git):
  READ: log, diff, status, blame, show, branch, shortlog, ls-files, ls-tree, describe, tag --list
  WRITE: clone, checkout, pull, add, commit, push, tag, stash, fetch, merge, rebase, cherry-pick, reset

Rules:
  - For write actions, ALWAYS use the confirmation format (see below)
  - For read actions, execute directly
  - Maximum 5 @git rounds per message
  - Maximum 3 @fetch rounds per message
  - Output truncation: 8000 chars for @git, 4000 chars for @fetch
```

### 1.2 AI Responds with Read Action

```
AI response: "Let me check the recent activity.
@git(log --since='1 week ago' --pretty=format:'%h %an %s' --no-merges)"

ChatViewModel:
  1. Receives streamed response from CLIBridge
  2. PromptEnrichmentService.detectGitCommands(response) finds @git(...)
  3. Extracts command: "log --since='1 week ago' --pretty=format:'%h %an %s' --no-merges"
  4. Validates against allowlist → "log" is allowed and is a READ command
  5. Executes: GitService.shared.executeRaw(command: "log ...", repoPath: gitContext.localPath)
  6. Gets output string
  7. Removes @git() from displayed message (user sees clean text)
  8. Sends result back to CLI as a follow-up context message:
     "[Git Result] Output of: git log --since='1 week ago'...\n<output here>"
  9. AI receives result and generates human-readable summary
  10. User sees: "This week there were 12 commits. The main changes were..."
```

### 1.3 AI Responds with Write Action (Confirmation Flow)

This is the key difference. Write actions do NOT execute automatically. They trigger a confirmation card.

```
AI response: "I'll create a new branch for this feature.
@git-confirm(checkout -b feature/user-auth, title=Create Branch, details=Branch 'feature/user-auth' from 'main')"

ChatViewModel:
  1. PromptEnrichmentService.detectGitConfirmations(response) finds @git-confirm(...)
  2. Parses: command="checkout -b feature/user-auth", title="Create Branch", details="..."
  3. Creates a PendingGitAction model:
     PendingGitAction(
       id: UUID(),
       type: .localGit,
       command: "checkout -b feature/user-auth",
       title: "Create Branch",
       details: ["Repository": "acme-api", "Branch": "feature/user-auth", "From": "main"],
       status: .pendingConfirmation
     )
  4. Inserts a GitActionConfirmationCard into the chat message stream
  5. User sees the confirmation card inline in the chat
  6. User clicks "Confirm":
     → GitService.shared.executeRaw(command, repoPath) runs
     → Card updates to show "Completed" with green checkmark
     → Result is sent back to AI as context
  7. User clicks "Cancel":
     → Card updates to show "Cancelled"
     → AI is informed: "[Git Action Cancelled] User declined: checkout -b feature/user-auth"
```

Same flow for platform write actions:

```
AI response: "I'll create an issue to track this bug.
@fetch-confirm(github.create_issue, repo=acme-api, title=Login fails on Safari, labels=bug,priority-high, body=Users report...)"

ChatViewModel:
  1. Detects @fetch-confirm(...)
  2. Creates PendingGitAction with type: .platformAPI
  3. Shows confirmation card with issue preview
  4. On confirm: ConnectorExecutor executes the action
  5. On cancel: AI is informed
```

### 1.4 New Intercept Syntax Summary

| Syntax | Type | Confirmation | Example |
|--------|------|-------------|---------|
| `@git(...)` | Local read | No (auto-execute) | `@git(log -10 --oneline)` |
| `@git-confirm(command, title=..., details=...)` | Local write | Yes (card shown) | `@git-confirm(push origin main, title=Push to Remote, details=Push 3 commits to origin/main)` |
| `@fetch(connector.action, params...)` | Platform read | No (auto-execute) | `@fetch(github.list_prs, repo=acme-api, state=open)` |
| `@fetch-confirm(connector.action, params...)` | Platform write | Yes (card shown) | `@fetch-confirm(github.create_issue, repo=acme-api, title=...)` |

### 1.5 Multi-Step Chains

The AI can chain multiple operations. Each write operation pauses for confirmation before continuing. Example:

```
User: "Create a release branch, tag it as v2.0, and create a GitHub release"

AI: "I'll do this in three steps.

Step 1: Create the release branch.
@git-confirm(checkout -b release/2.0, title=Create Release Branch, details=Branch 'release/2.0' from 'main')

[User confirms → executes → result sent back to AI]

AI: "Branch created. Step 2: Tag it.
@git-confirm(tag -a v2.0 -m 'Release 2.0', title=Create Tag, details=Tag 'v2.0' on branch 'release/2.0')

[User confirms → executes → result sent back to AI]

AI: "Tagged. Step 3: Push and create the GitHub release.
@git-confirm(push origin release/2.0 --tags, title=Push Branch and Tags, details=Push release/2.0 and tag v2.0 to origin)

[User confirms → executes → result sent back to AI]

AI: "Pushed. Now creating the GitHub release.
@fetch-confirm(github.create_release, repo=acme-api, tag=v2.0, name=Release 2.0, body=## What's New\n- Feature A\n- Feature B\n- Bug fix C, generate_notes=true)

[User confirms → executes]

AI: "Done. Release 2.0 is live at https://github.com/joseconti/acme-api/releases/tag/v2.0"
```

## 2. Contextual AI Actions on Every UI Element

Every visible element in the Git section must have an "Ask AI" action. This is implemented as context menu items and/or inline buttons that pre-populate the chat with a specific prompt.

### 2.1 Pull Request Actions

Each PR row in `GitPRListView` gets a context menu (right-click) and an inline icon button:

| Action | What it does | Pre-filled chat prompt |
|--------|-------------|----------------------|
| Review PR | AI downloads diff and does a full code review | "Review PR #{{number}} '{{title}}' in detail. Analyze the diff for bugs, security issues, performance problems, and style inconsistencies. Be specific about line numbers." |
| Summarize PR | AI explains what the PR does in plain language | "Summarize what PR #{{number}} '{{title}}' does. Explain the changes in non-technical terms." |
| Suggest improvements | AI proposes code improvements | "Look at PR #{{number}} '{{title}}' and suggest concrete improvements to the code. Focus on readability, maintainability, and edge cases." |
| Check for conflicts | AI checks if the PR can merge cleanly | "Check if PR #{{number}} from '{{sourceBranch}}' to '{{targetBranch}}' has merge conflicts. If so, explain what conflicts exist and how to resolve them." |
| Post review comment | AI writes and posts a review | "Write a review comment for PR #{{number}} and post it. Be constructive and specific." |
| Merge PR | AI merges (with confirmation) | "Merge PR #{{number}} into {{targetBranch}}." |

## 3. Intelligent AI Features (The Breakthrough Stuff)

These are features that no other Git client offers because they require an AI.

### 3.1 "Explain This Repo" — AI Codebase Onboarding

**Trigger**: Button in repo header, or user asks "explain this repo".

**What the AI does (multi-step)**:

1. Fetches the file tree (`@fetch(github.get_contents, path=/)`)
2. Reads key files: README.md, package.json / Package.swift / Cargo.toml / pyproject.toml, Dockerfile, CI configs
3. Analyzes directory structure to identify patterns (MVC, MVVM, monorepo, microservices...)
4. Generates a comprehensive onboarding document:

```
## Repository: acme-api

### Overview
REST API for the Acme platform, built with Python 3.11 and FastAPI.

### Technology Stack
- Language: Python 3.11
- Framework: FastAPI 0.104
- Database: PostgreSQL 15 via SQLAlchemy 2.0
- Cache: Redis
- Testing: pytest with 87% coverage
- CI/CD: GitHub Actions → AWS ECS

### Architecture
Three-layer architecture:
- routes/ — HTTP endpoint definitions (23 route files)
- services/ — Business logic (18 service classes)
- repositories/ — Database access (15 repository classes)

### Key Entry Points
- main.py — App initialization and middleware setup
- routes/__init__.py — Route registration
- config.py — Environment-based configuration

### How to Run
1. docker-compose up -d (PostgreSQL + Redis)
2. pip install -r requirements.txt
3. uvicorn main:app --reload

### Active Development
- 5 contributors in the last month
- 12 open PRs (3 need review)
- Most active areas: routes/auth, services/billing
```

**Why this is rompedor**: Joining a new project usually takes days of reading docs and code. This gives you a full understanding in 30 seconds. No Git client does this.

### 3.2 AI-Powered PR Review

**Trigger**: "Review PR" button on any PR row, or user asks "review PR #15".

**What the AI does (multi-step)**:

1. Fetches PR metadata: `@fetch(github.list_prs, number=15)` → gets title, description, author, branch
2. Fetches the full diff: `@fetch(github.get_pr_diff, number=15)` → gets all changed files with line-by-line diff
3. For each changed file, the AI analyzes:
   - Logical correctness (does the code do what the PR says it does?)
   - Security issues (SQL injection, XSS, hardcoded secrets, auth bypasses)
   - Performance (N+1 queries, unnecessary allocations, missing indexes)
   - Style consistency (does it follow the project's conventions?)
   - Edge cases (what happens with nil, empty arrays, network failures?)
   - Missing tests (are new code paths covered?)
4. Generates a structured review:

```
## PR Review: #15 "Add user authentication"

### Summary
This PR adds JWT-based authentication with login, logout, and token refresh endpoints.
14 files changed, 487 additions, 23 deletions.

### Issues Found

**[CRITICAL] Hardcoded JWT secret (auth/config.py:12)**
The JWT signing secret is hardcoded as "my-secret-key". This must be loaded from environment variables.

**[BUG] Token expiry not checked on refresh (auth/service.py:67)**
The refresh endpoint doesn't verify that the refresh token hasn't expired. An expired token can be used to generate new access tokens indefinitely.

**[PERFORMANCE] N+1 query in user permissions (auth/repository.py:34)**
Each permission check makes a separate DB query. This should be a single JOIN query.

**[STYLE] Inconsistent error responses (auth/routes.py:28,45,62)**
Some endpoints return {"error": "..."} and others return {"detail": "..."}. The rest of the API uses "detail".

**[MISSING TEST] No test for token refresh with expired token**
The happy path is tested but the expired token edge case is not.

### Verdict
The PR needs changes before merge. The hardcoded secret and token expiry issues are critical.
```

5. **Optional**: The user can say "post this review" and the AI posts it as a comment on the PR via `@fetch-confirm(github.create_comment)`.

### 3.3 "What Broke?" — AI Regression Detective

**Trigger**: Button in repo header, or user asks "something broke, help me find what".

**What the AI does**:

1. Asks the user: "When did it start failing? What's the symptom?"
2. Fetches recent commits: `@git(log --since='3 days ago' --pretty=format:'%H %an %s' --no-merges)`
3. For each commit, fetches the diff: `@git(show --stat <hash>)` to see which files changed
4. Analyzes which commits touch "risky" areas:
   - Configuration files (config, env, CI)
   - Database migrations
   - Authentication/authorization code
   - Core business logic
   - Dependencies (package.json, Gemfile, requirements.txt)
5. Ranks commits by "regression probability" and presents findings:

```
## Regression Analysis

Based on the symptom "login page returns 500 error" and recent commits:

### Most Likely Cause
**Commit abc1234** by @maria — "Refactor user model"
- Changed: models/user.py, services/auth.py, migrations/0045_alter_user.py
- This commit renamed the `email_verified` field to `is_verified` in the User model
  but the auth service still references `email_verified` on line 23.
- The migration ran successfully but the code wasn't updated everywhere.

### Also Investigate
**Commit def5678** by @pedro — "Update SQLAlchemy to 2.1"
- Dependency update that could have breaking changes.

### Unlikely but Changed Auth Area
**Commit 789abc** by @jose — "Add rate limiting to login endpoint"
- Only adds middleware, doesn't touch auth logic.

### Suggested Fix
In services/auth.py line 23, change `user.email_verified` to `user.is_verified`.
```

### 3.4 Automatic Changelog & Release Notes Generation

**Trigger**: "Generate changelog" button or user asks "generate changelog since v1.5".

**What the AI does**:

1. Gets the last tag: `@git(describe --tags --abbrev=0)`
2. Gets all commits since that tag: `@git(log v1.5..HEAD --pretty=format:'%H|%an|%s' --no-merges)`
3. For each commit, determines the category by analyzing the message AND the diff
4. Cross-references with closed issues and merged PRs: `@fetch(github.list_issues, state=closed, since=<tag_date>)`
5. Generates formatted release notes:

```
## v2.0.0 Release Notes

### New Features
- **User authentication** — JWT-based login, logout, and token refresh. (#15, @maria)
- **Rate limiting** — Configurable rate limits on all API endpoints. (#18, @jose)
- **Export to CSV** — Users can now export their data in CSV format. (#22, @pedro)

### Bug Fixes
- Fixed pagination returning duplicate results on the last page. (#20, @maria)
- Fixed WebSocket disconnection when the server restarts. (#24, @jose)

### Breaking Changes
- The `email_verified` field on the User model has been renamed to `is_verified`. Update any code that references this field directly. (#15)
- The `/api/v1/users` endpoint now requires authentication. (#18)

### Internal
- Upgraded SQLAlchemy from 2.0 to 2.1.
- Refactored database connection pooling for better performance.

### Contributors
@maria (5 commits), @jose (4 commits), @pedro (2 commits)
```

6. The user can then say "create the release" and the AI does:
   `@git-confirm(tag -a v2.0.0 -m 'Release 2.0.0', ...)`
   `@git-confirm(push origin --tags, ...)`
   `@fetch-confirm(github.create_release, tag=v2.0.0, name=v2.0.0, body=<changelog>, ...)`

### 3.5 AI Commit Assistant

**Trigger**: User asks "commit my changes" or clicks a "Commit" button in the UI.

**What the AI does**:

1. Checks status: `@git(status --porcelain)`
2. Shows the user what's changed: `@git(diff --stat)` and `@git(diff)` for details
3. Analyzes the changes and generates a commit message following the project's conventions:
   - Detects if the project uses Conventional Commits, Angular style, etc. by analyzing `@git(log -20 --pretty=format:'%s')`
   - Generates a message that matches the detected pattern
4. Presents the commit plan:

```
@git-confirm(add src/auth/service.py src/auth/routes.py tests/test_auth.py,
  title=Stage Files,
  details=Stage 3 modified files: service.py, routes.py, test_auth.py)
```

After staging confirmation:

```
@git-confirm(commit -m 'fix(auth): check token expiry on refresh\n\nEnsure refresh tokens are validated for expiration before issuing\nnew access tokens. Adds test coverage for expired token scenario.\n\nCloses #24',
  title=Create Commit,
  details=Commit message follows Conventional Commits format. Closes issue #24.)
```

5. Optionally pushes: "Want me to push this to the remote?"

### 3.6 Automated Repo Monitoring (Integration with Schedules)

**Trigger**: User sets up a schedule from the Git section, or asks "monitor this repo daily".

This integrates the Git section with the existing Schedules system (doc 10). The user can create scheduled tasks that monitor their repos.

**Examples of scheduled monitors**:

| Monitor | Schedule | What it does | Notification |
|---------|----------|-------------|-------------|
| PR Reviewer | Every morning at 9:00 | Lists PRs that need review, warns about PRs older than 3 days | "You have 3 PRs waiting for review. PR #15 has been open for 5 days." |
| CI Watcher | Every 30 minutes | Checks if any CI pipeline has failed on main | "CI failed on main after commit abc1234 by @maria." |
| Stale Branch Cleaner | Every Monday | Lists branches with no activity in 30+ days | "5 branches have had no activity in 30+ days: feature/old-thing, ..." |
| Security Scan | Every Sunday | Checks for known vulnerabilities in dependencies | "3 dependencies have known vulnerabilities. Update lodash, express, axios." |
| Activity Summary | Every Friday at 17:00 | Weekly development activity report | "This week: 23 commits, 8 PRs merged, 5 issues closed. Most active: auth module." |
| Issue Triage | Daily at 10:00 | New unassigned issues | "3 new issues have no assignee: #45, #46, #47." |

**Implementation**: Each monitor is a `CronJob` with `connectorBindings` pointing to the GitHub/GitLab connector. The payload contains the analysis prompt. The AI processes the data and delivers results via the configured channel (notification, Slack, Telegram, etc.).

**UI**: In the repo header, a "Monitor" button opens a sheet with pre-configured monitor templates. The user picks one, adjusts the schedule, and saves. This creates a CronJob linked to the repo.

### 3.7 Cross-Repo Intelligence

**Trigger**: User has multiple repos selected/connected and asks about relationships between them.

**What the AI does**:

When the user asks "what do I need to change in gateway if I update the WebSocket protocol in mcclaw?":

1. AI reads the relevant files from both repos (via file tree + content fetching)
2. Identifies the interface/contract between them (e.g., WebSocket message format, API endpoints)
3. Maps dependencies: which files in repo A depend on which files in repo B
4. Generates a coordinated change plan:

```
## Cross-Repo Impact Analysis

### Change: Update WebSocket protocol from v3 to v4 in McClaw

### McClaw (source of change)
Files to modify:
- Services/Gateway/GatewayConnectionService.swift (protocol version constant)
- Models/Gateway/GatewayModels.swift (new message types)

### Gateway (dependent repo)
Files that need updating:
- src/ws/protocol.ts (version check, new message handlers)
- src/ws/types.ts (matching type definitions)
- tests/ws/protocol.test.ts (test cases for new messages)

### Suggested Order
1. Update Gateway first (add v4 support while keeping v3 backward compatible)
2. Update McClaw to send v4 messages
3. Remove v3 support from Gateway after all clients upgrade
```

### 3.8 AI-Assisted Merge Conflict Resolution

**Trigger**: User does a pull/merge and gets conflicts, or asks "help me resolve these conflicts".

**What the AI does**:

1. Detects conflicts: `@git(status)` → finds files with "UU" (unmerged) status
2. For each conflicted file: `@git(show :1:<file>)` (base), `@git(show :2:<file>)` (ours), `@git(show :3:<file>)` (theirs)
3. Analyzes the intent of both changes (using commit messages and diffs for context)
4. Proposes a resolution for each conflict, explaining the reasoning:

```
## Conflict Resolution: src/auth/service.py

### Conflict at lines 23-31

**Your branch (feature/auth)** changed the token validation logic to check expiry.
**Their branch (main)** added logging to the same function.

### Both changes are complementary and should be kept.

Proposed resolution:
```python
def validate_token(self, token: str) -> bool:
    logger.info(f"Validating token for user {token.sub}")  # from main
    if token.exp < datetime.utcnow():                       # from feature/auth
        logger.warning(f"Expired token for user {token.sub}") # merged
        return False
    return self._verify_signature(token)
```

Shall I apply this resolution?
```

5. On confirmation, writes the resolved file and stages it:
   `@git-confirm(add src/auth/service.py, title=Apply Resolution, details=Apply AI-suggested merge resolution for service.py)`

**Note**: This is NOT McClaw acting as a full IDE merge tool. It's the AI understanding the semantic intent of both sides and proposing intelligent resolutions. The user confirms each one. For complex conflicts, the AI suggests opening the file in the user's IDE.

## 4. New Data Models

### 4.1 PendingGitAction

```swift
struct PendingGitAction: Identifiable {
    let id: UUID
    let type: GitActionType
    let command: String               // git command or connector action
    let title: String                 // Human-readable title for the card
    let details: [String: String]     // Key-value pairs shown in the card
    var status: GitActionStatus

    enum GitActionType {
        case localGit                 // Executed via GitService
        case platformAPI              // Executed via ConnectorExecutor
    }

    enum GitActionStatus {
        case pendingConfirmation      // Waiting for user to confirm/cancel
        case executing                // Running the command
        case completed(output: String) // Done successfully

## 9. Prompt Engineering Notes

The quality of all these features depends on good prompts. Key principles:

1. **Always include the full Git context header** — the AI must know the repo, branch, platform, and available actions.
2. **Be specific in prompt templates** — "Review this PR" is vague. "Review PR #15. Analyze the diff for bugs, security issues, performance problems, and style inconsistencies. Be specific about line numbers." is actionable.
3. **Tell the AI what format to use** — For reviews, specify "use markdown headers for each issue found, prefix severity as [CRITICAL], [BUG], [STYLE], [SUGGESTION]".
4. **Include project conventions when possible** — If the commit messages follow Conventional Commits, tell the AI.
5. **Limit scope to avoid hallucination** — "Analyze the diff of this PR" is better than "tell me everything about this code" because it constrains the AI to actual data.
6. **Chain operations explicitly** — When multiple steps are needed (changelog → tag → release), the AI should do one step at a time, waiting for confirmation between write operations.

## 10. Security Considerations

1. **All write operations require confirmation** — No exceptions. The AI cannot push, commit, merge, create issues, or modify anything without the user clicking "Confirm".
2. **Command allowlist** — The `@git()` intercept only executes commands from the allowlist in GitService. Commands like `git config`, `git filter-branch`, `git reflog`, or any command with `--force` are blocked.
3. **Destructive operation warnings** — Commands like `reset`, `rebase`, `push --force` show an additional warning in the confirmation card: "This is a destructive operation that may lose data."
4. **Sensitive content filtering** — Before displaying @git results in the chat, scan for patterns that look like secrets (API keys, tokens, passwords). If found, redact and warn the user.
5. **Rate limiting** — Maximum 5 @git rounds and 3 @fetch rounds per AI response. This prevents runaway AI loops.
6. **Local path validation** — GitService only operates on paths that match known local repos (previously discovered or manually configured). No arbitrary filesystem access.
